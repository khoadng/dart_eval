вқҜ what are the better way? why this contract is $Value                          

вҸә The reason is that $Value serves two purposes at once, and they should be
  separated.

  Purpose 1: Type-uniform dispatch. When eval'd code calls .length on something,
   the runtime doesn't know at that point if it's a $String, $List, or an eval'd
   class. $getProperty(runtime, 'length') works on anything. Raw Object? can't
  do that -- you'd need if (x is String) ... else if (x is List) ....

  Purpose 2: Eval'd subclass detection. When a bridge class is subclassed in
  eval'd code, $Bridge.$getProperty checks if the property was overridden. This
  needs the $Value/$Instance interface to delegate to the eval'd subclass.

  But here's the thing -- neither purpose requires wrapping primitives. Nobody
  subclasses int in eval'd code. Nobody overrides int.+ in a script. Primitives
  only need $Value wrapping because the bridge signature demands it uniformly.

  The better design splits the contract:

  // Primitives flow as raw Dart values -- int, double, bool, String, null
  // Only eval'd instances and bridge objects get wrapped

  // Instead of one universal type:
  typedef EvalCallableFunc = $Value? Function(Runtime, $Value?, List<$Value?>);

  // Two-tier approach:
  typedef BridgeFunc = Object? Function(Runtime, Object?, List<Object?>);
  // ^-- args come in as raw values (int, double, String, $Instance, etc.)
  // ^-- return raw values, runtime boxes only if needed

  The bridge function for int.abs() becomes:

  // Current: 4 allocations (unbox target, unbox args, compute, box result)
  $Value? _abs(Runtime rt, $Value? target, List<$Value?> args) {
    return $int((target!.$value as int).abs());
  }

  // Better: 0 allocations
  Object? _abs(Runtime rt, Object? target, List<Object?> args) {
    return (target as int).abs();
  }

  The InvokeDynamic op would do the type dispatch that $getProperty does today
  -- but at the op level, not the value level. For eval'd instances it checks
  the vtable. For bridge objects it calls the bridge handler. For raw

  The 223 stdlib functions would all get simpler (no $value unwrapping, no
  $int() re-wrapping), and primitives never allocate.
